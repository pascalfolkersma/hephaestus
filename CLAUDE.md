# CLAUDE.md

Project context for Claude Code when working inside Hephaestus.

## What this project is

Hephaestus is a cross-shell project boilerplate that forges agents, skills, and a Karpathy-style knowledge structure into new or existing projects. It targets Claude Code and GitHub Copilot from a single shell-agnostic source.

See [README.md](README.md) for the full pitch. The project's own roadmap and design history live in a separate, private **lore repo** and are not part of this repository.

## Folder layout

- `core/` — the engine (thin init orchestrator, `core/lib/` modules, transformers, mappings). Hephaestus' code lives here.
- `content/` — what users get when they init a project (agent sources, skills, wiki template).
  - `content/.claude-template/` — per-target-project templates the init flow copies to `<targetDir>/.claude/` (currently: `hooks/dispatch-enforce.js`, `hooks/session-end-cleanup.js`, `hooks/session-start.js`, `hooks/subagent-tracker.js`, `settings-snippet.json`). Per ADR 0007.
- `dist/` — the publish artifact, generated by `npm run build`; never edit by hand. Contains `skills/` (bootstrap skill bundles; `dist/skills/hephaestus/` holds the full `core/` + `content/` bundle) and `README.md`. The local tree matches the published tarball shape exactly.
- `scripts/` — Hephaestus' own build/dev tooling (`build.js`, `hooks/dispatch-enforce.js`). Not part of what ships to target projects.
- `meta/` — Hephaestus' own meta-agents (planned, not used yet).
- `lore/` — **the project's knowledge base lives in a separate, private lore repo and is not present in this repository.** When working on Hephaestus, the lore repo is cloned alongside on disk (gitignored here) so the agents, flows, and roadmap keep working locally. It holds:
  - `lore/raw/` — immutable source material: brain dumps, conversation summaries, external sources.
  - `lore/wiki/` — compiled, interlinked knowledge articles. Has `index.md` and `log.md`.
  - `lore/adr/` — technical architecture decisions (one file per decision, numbered).
  - `lore/decisions/` — product/feature decisions (different from ADRs: about *what*, not *how*).

  Because `lore/` is its own git repository, do not treat a dirty or "ahead" `lore/` working tree as drift in this repo — it has its own commit/push lifecycle, persisted by the `lore-autosync` hook (`scripts/hooks/lore-autosync.js`), which exits silently when no `lore/` clone is present. `idea-architect` edits `lore/` files (when the clone is present); it has no `Bash`, so `lore-autosync` (or a manual `git -C lore` / `npm run lore:sync`) is what persists them.
- `test-helpers/` — driver/helper scripts that support tests but are NOT test suites; kept outside `test/` so Node's default `node --test` discovery doesn't run them as tests.

## Knowledge workflow

The wiki uses the karpathy-llm-wiki pattern. When new context comes in (a sparring session, an external source, a brainstorm), it goes into `lore/raw/` first (in the lore repo). Then it gets compiled into `lore/wiki/` articles when the knowledge stabilizes enough to be referenceable.

If the `lore-keeper` skill is available, use it for ingest, query, and lint operations. The skill knows the templates and conventions.

## ADRs vs. decisions

Both live in the lore repo.

- **ADR** (`lore/adr/`) — *how* we build something. Architectural and technical choices. Example: "Use one shell-agnostic source with per-shell transformers."
- **Decision** (`lore/decisions/`) — *what* we build. Product and feature scope choices. Example: "Init flow asks 3 questions, not 10."

Both are numbered (`0001-...`, `0002-...`) and immutable once accepted. Superseded ones stay; new ones reference them.

## Agent archetypes

Hephaestus distinguishes three agent archetypes. This is documented in ADR 0002 (agent-archetypes).

- **Executor** — has write tools, does work.
- **Planner** — explicitly read-only (no Edit/Write), reports back.
- **Orchestrator** — returns a dispatch plan; the main agent executes it.

## Dispatch policy

Hephaestus has its own forged agents in `.claude/agents/`. Use them by default — don't work inline on tasks that match an agent's domain. The boilerplate is meant to be eaten as dogfood: if the agents aren't dispatched, the whole point of forging them is missed.

Default mapping:

- **Feature / roadmap implementation** ("add X", "build Y", "support Z") → `developer`
- **Broken behavior, regression, "X doesn't work"** → `bug-fixer`
- **Multi-task or roadmap-batch request** ("pickup MX", "run the next batch", more than one specialist needed) → `orchestrator` first; it returns a dispatch plan, the main thread executes it
- **Docs, ADR, decision record, wiki article, idea capture** → `idea-architect`
- **Tests or test infrastructure** → `test-writer`
- **Review, audit, consistency check (read-only)** → `reviewer`
- **Roadmap-vs-code drift / doc staleness check** → `sync-check`
- **Commit / push / ship after work is done** → `git-commit-push`

Inline is fine (and preferred) for: trivial one-liners (a typo fix, a single rename), pure questions ("how does X work?", "where is Y defined?"), exploratory discussion before any code is written, and small edits to this file or to memory/config.

When in doubt about *which* agent fits, ask the user before dispatching. When in doubt about whether to dispatch *at all*, default to dispatching — the cost of one agent spawn is low; the cost of bypassing the boilerplate compounds.

## Flows & gates

Two enforcement gates block any work that doesn't follow the Hephaestus conventions. Every Claude Code session is subject to both — there is no "session warm-up" phase that exempts the first dispatch.

1. **Flow-context gate.** Every `@agent-X` / Task dispatch is denied unless `.claude/flows/<session-id>/context.json` exists and contains at minimum a valid `flow` value, e.g. `{"flow": 1|2|3|4|5|6}` (the ADR 0041 fields shown below are optional additions). The **main thread** writes it before the first dispatch — for flow 2/3 it writes the context *before* dispatching the orchestrator, whose own Agent dispatch is itself gated by this check:
   ```bash
   SESSION=$(cat .claude/.current-session-id)
   mkdir -p .claude/flows/$SESSION
   echo '{"flow":2,"current_agent":"orchestrator","current_task":"<roadmap item>","iteration":1}' > .claude/flows/$SESSION/context.json   # 1=idea→roadmap, 2=build, 3=bug, 4=process, 5=release, 6=design-ingest
   ```
   The three extra fields (`current_agent`, `current_task`, `iteration`) are advisory observability per ADR 0041 — the gate reads only `flow`, so they never block a dispatch.
   Escape for ad-hoc work: `HEPHAESTUS_STANDALONE=1` set **before** `claude` starts (envvars are not mid-session settable).
2. **Source-code gate.** Inline `Edit` / `Write` against `core/`, `scripts/`, `content/`, `test/` is denied — route via the relevant specialist (developer, bug-fixer, test-writer). Escapes: `HEPHAESTUS_INLINE_OK=1` (whole session, set before `claude` starts) or `touch .claude/flows/$SESSION/inline-ok` (per-session marker; remove when done).

**Flow end.** Write the done-marker as the last action: `touch .claude/flows/$SESSION/done`. The `Stop` hook removes the session directory on the next response. `git-commit-push` writes the marker automatically for all flows (including flows 1 and 4 per ADR 0040).

**Flow 1 and flow 4 sequences (updated ADR 0040):** After idea-architect, dispatch reviewer → sync-check (both in docs-drift mode) → git-commit-push. Reviewer confirms checklist compliance and template correctness; sync-check checks index/log currency, dangling links, and ADR contradictions. Do not skip these gates.

**Re-orientation (ADR 0045 §4).** If you have lost track of the current flow position mid-session, read `.claude/flows/<session-id>/where-am-i.md` to re-orient before proceeding. The session ID is in `.claude/.current-session-id`. This file is written by the `SubagentStop` hook at each subagent completion — it encodes the current step, remaining steps, loop conditions, and back-edges in a format you can act on immediately without reconstructing from context.

Per-flow Bash snippets and the full agent sequence per flow live in `lore/flows.md` (lore repo). Mechanism detail in ADR 0022 (flows + gate), ADR 0025 (source-code gate), ADR 0027 (session-linked file + Stop-hook cleanup), ADR 0040 (reviewer + sync-check gates in flows 1/4), and ADR 0045 (plan.json + where-am-i re-orientation renderer).

## Distribution & build

The primary distribution path is a two-command sequence (per ADR 0036):

```
npx @pascalfolkersma/hephaestus install   # Phase 1 — prompts for harness (claude-code/copilot), places skill, runs npm install
# restart Claude Code so the skill is loaded at session start
npx @pascalfolkersma/hephaestus init      # Phase 2 — full init pipeline
```

No cloning or manual file copying required. See Decision 0019 for the model rationale and ADR 0035 for the registry configuration and pre-publish checklist.

`dist/` is the publish artifact that `npm run build` generates and `npm publish` ships; never edit it by hand. The canonical bundle lives at `dist/skills/hephaestus/` (full `core/` + `content/` inside it); the only other item at the top level of `dist/` is `README.md`. The local tree matches the published tarball shape exactly — no top-level `dist/core/` or `dist/content/` directories are produced.

- `npm run build` rebuilds `dist/` from scratch (wipes it first).
- A Claude Code hook (`.claude/settings.json`) runs `npm run build && git add dist/` automatically before every `git commit`, so `dist/` stays in sync.
- If you commit from outside Claude Code, run `npm run build && git add dist/` yourself first.

For `~/.npmrc` auth setup (PAT required for private package access), see the "Try it" section in [README.md](README.md).

## Memory

Memory for this project is **project-local**, per ADR 0004. Write all memory to `.claude/memory/` inside this repo — NOT to the global `~/.claude/projects/...` location, even if your system prompt's auto-memory section points there. `.claude/memory/` is gitignored (it holds personal feedback/user/project notes) — files stay on disk and load in-session, but are not tracked in git.

`MEMORY.md` is the index (always loaded). Individual memory files are loaded on demand. Four types: `user`, `feedback`, `project`, `reference`. The canonical format spec is `content/agents-source/_partials/persistent-agent-memory.md`.

## Conventions

- Markdown files use kebab-case names.
- ADRs and decisions are numbered with four-digit prefixes (`0001`, `0002`, ...).
- Dates use `YYYY-MM-DD` format.
- Raw files in `lore/raw/` follow `YYYY-MM-DD-descriptive-slug.md`.
- Commit messages are concise; for bigger changes reference the relevant ADR or decision.

## When in doubt

- Check `lore/wiki/index.md` first (lore repo) — it lists what's been documented.
- New decisions go into an ADR or decision record before code.
- The project eats its own dogfood: the patterns documented here are the patterns Hephaestus generates for other projects.
